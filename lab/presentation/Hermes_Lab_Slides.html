What You'll Build Today

The eighth member of the Hermes team:

The Optimization Agent

  • 3+ skill definitions — the building blocks
  • One complete agent — README + system prompt
  • One workflow — wiring it into the runner

Everything is Markdown and JSON. Not a line of Python.

What Hermes Is

  • A Clinical Supply Chain AI Agent Team for pharmaceutical trial supply management
  • Seven specialised agents collaborate — ingest data, analyse demand, plan supply, monitor risk, enforce compliance, report out
  • System-agnostic: reads CSV/JSON data drops, not direct ERP/RTSM connections
  • Portable: the same study package structure works for any protocol
  • Intelligence lives in plain Markdown, not code
Educational/exploratory tool only — not a validated system.

Three Guiding Principles

1. Plan top-down, build bottom-up

Read: workflow → agent → skill. Build: skill → agent → workflow.

2. Markdown is the code

If you can write a clear instruction, you can write an agent.

3. Errors are the lesson

When something fails, read the error. Don't just re-run.

MODULE 1 · 20 min

Architecture Orientation

A mental map of Hermes

M1 · Purpose

Get oriented with what's already in the starter kit.

Goal: know where things live, so when you build your own agent the surrounding landscape makes sense.

The 5 Layers of Hermes

Layer 5: RUNNER / COMMANDS  → executes workflows
Layer 4: WORKFLOWS          → which agents run, in what order
Layer 3: AGENTS             → AI specialists with their own instructions
Layer 2: SKILLS             → atomic capabilities agents apply
Layer 1: STUDY PACKAGE     → your study's config + CSV data

Runner orchestrates · workflows sequence · agents specialise · skills do one thing · study package is the data.

Folder Map

c:\Hermes\
├── README.md                Quickstart
├── CLAUDE.md                Claude Code config
├── Hermes_Grimoire.md       The encyclopedia
│
├── agents/                  7 specialised agents
├── skills/                  76 shared skill definitions
├── workflows/               5 workflow markdowns + workflows.json
├── runner/                  Python execution engine
├── studies/                 Per-study packages
├── StudyDataCreation/       Browser-based test-data generator
├── lab/                     3-hour workshop kit
└── .claude/                 Commands, settings

Two Key Truths

Skills are a shared library

Not agent-owned. The same skill (e.g. SI-01) can be called by several agents. This keeps the system small.

Every agent has the same two-file shape

README.md + system_prompt.md. Learn the shape once, apply it seven times.

What Version Are You Building?

Today: Prompt-Only

  • Every agent & skill = a Markdown file
  • No custom code
  • Accessible to non-technical users
  • Easy to inspect & modify

Production: Agent SDK

  • Each agent gets an autonomous runtime
  • Agents call each other directly
  • Connect to live data via MCP — no data drops
  • End-to-end with no human in the loop

The Markdown you write today becomes the system prompts for the production version.

MODULE 2 · 50 min

Configure Your Study

Build the data foundation · run a baseline

M2 · Purpose

Create a real, working study package from your Scenario Card, generate test CSV data, and run a baseline end-to-end chain.

By the end: a functioning Hermes study and output files to compare your own agent against later.

Key Concept · System-Agnostic

  • Hermes does not talk to ERP or RTSM systems directly
  • It reads CSV/JSON data drops that any source can produce
  • That's what makes it portable across studies and sponsors
  • The data layer is the boundary — change the source, reuse the intelligence

The Study Package Shape

studies/<STUDY_ID>/
├── config/
│   ├── study_config.json      arms, items, countries, sites
│   ├── supply_network.json    depots, manufacturers, lanes
│   └── policies.json          safety stock, reorder, budget
├── reference/
├── data_drops/
│   └── 2026-04-22/            ← dated folder (YYYY-MM-DD)
│       ├── ctms_plan.csv
│       ├── erp_inventory.csv
│       └── rtsm_actuals.csv
└── outputs/

The date format isn't decorative — the runner uses it to find the right drop.

The Flow You'll Follow

  1. /init-study <STUDY_ID>  creates folder skeleton
  2. Open StudyDataCreation/index.html in your browser
  3. Fill in Study Config → download JSON
  4. Fill in Supply Network → download JSON
  5. Fill in Policies → download JSON
  6. Move all three into studies/<STUDY_ID>/config/
  7. Generate test CSVs → drop into data_drops/<today>/
  8. /preflight  integrity check
  9. /run-workflow-chain <STUDY_ID> WF-01 WF-02 WF-05

The Slash Commands

/init-study

Creates the study folder skeleton with templates.

/preflight

Integrity gate — catches missing files, malformed JSON, broken references before they become confusing agent errors 20 minutes later.

/run-workflow-chain

Runs multiple workflows in sequence. Today's baseline: WF-01 → WF-02 → WF-05 (demand → supply → execution).

Commands live in .claude/commands/. They're part of the framework, not magic.

⚠ The #1 Time Sink

ID Mismatches

  • ARM-A vs ARM_A ❌
  • SITE-01 vs Site-01 ❌
  • Item IDs in CSVs that don't match IDs in study_config.json ❌
If you get a weird error in Module 7, 90% of the time it's an ID typo from Module 2.

Use short, consistent IDs. Copy-paste rather than retype.

M2 · Run Your Baseline Chain

/run-workflow-chain <STUDY_ID> WF-01 WF-02 WF-05

Watch the agents execute:

  1. WF-01 — Supervisor validates
  2. WF-02 — Demand & Supply analysts
  3. WF-05 — Execution / reporting

When asked "save outputs?" → say yes. These are your before-picture for M7's after-picture.

M2 · Tips

  • Scenario Card is a starting point, not a contract — adjust values that feel unrealistic
  • Use short, memorable IDs (SITE-01, not CLINICAL-SITE-BOSTON-MASS-001)
  • Downloads go to your browser's Downloads folder — move them manually
  • Closed the browser? Use Load Config buttons to re-import and keep editing
  • Run /preflight before and after any change
  • Before moving on: git add -A && git commit -m "lab baseline" — saves you if you break something later

MODULE 3 · 15 min

Meet the Optimization Agent

Read the brief · plan before you build

M3 · Purpose

Switch from "configure existing Hermes" to "design a new agent".

Read the brief, then plan your agent — on paper — before touching any file.

The Order Matters

Read → Plan → Build

  • The brief tells you what the agent should do
  • The design decisions of how are yours
  • There is no single correct answer — two students will design differently-shaped agents that both work
  • Planning on paper forces brevity and is faster than typing

The One-Sentence Test

Can you describe your agent's purpose in one sentence?

If you can't → the scope is too wide. Keep cutting.

Name your agent's purpose in one sentence before you name any skill.

M3 · Plan Your Agent

Open lab/optimization_agent_brief.md. Then answer on paper:

  1. How many skills do you think this agent needs?
  2. What should each skill analyse? (one sentence each)
  3. What data file does each skill read?
  4. What signal will your agent emit if it finds a problem?

There's a sweet spot around 3 skills. Start there.

MODULE 4 · 40 min

Write Your Skills

The building blocks of your agent

M4 · Purpose

Write at least three skill files.

Each skill should:

  • Read one input
  • Do one analysis
  • Return one kind of finding

What Is a Skill?

A reusable procedure that answers four questions:

  • What do I analyse?
  • What data do I read?
  • What steps do I follow?
  • What do I return?

The 7 Sections of a Skill File

  1. Purpose — what the skill does (2–3 sentences)
  2. Owner — which agent uses this skill
  3. When This Skill Is Used — workflow & trigger
  4. Inputs — data files, config values, upstream outputs
  5. Steps — numbered procedure (4–6 steps)
  6. Output — what the skill returns
  7. Notes — caveats, limitations (optional)

Template: lab/templates/skill_template.md

Naming & Placement

/skills/OA-01_your_skill_name.md
        ↑  ↑
        │  └─ your first skill
        └──── OA = Optimization Agent (your category)

Use verbs: detect_, flag_, calculate_, identify_, spot_

Verbs force single responsibility. Nouns hide multiple jobs.

Suggested Focus Areas

#FocusData file
OA-01Depots above max-weeks policyerp_inventory.csv + policies.json
OA-02Lots likely to expire unusederp_inventory.csv
OA-03Sites with stock but no dispensingsite_inventory.csv
OA-04Lanes with repeated emergency shipmentserp_inventory.csv
OA-05Turn findings into recommendationsoutputs of other skills

Starting points — rename, reshape, or replace any of them.

Single Responsibility

If a skill name contains "and", it's two skills.

"Detect near-expiry lots and recommend action" → split into:

  • OA-02: flag_near_expiry_lots
  • OA-05: generate_recommendations

One skill, one CSV input. Crossing files usually means the skill is two skills.

What Good Looks Like

A well-written skill produces a finding a human can act on in one sentence:

"Depot APAC-01 holds 14 weeks of ARM-A kit stock against an 8-week target — 6 weeks excess, flag as OPTIMIZATION_OPPORTUNITY."

If your skill's output can't be phrased this crisply → scope is probably too broad.

Build One, Then Test It

  1. Write OA-01
  2. Ask Claude: "Read skills/OA-01 and tell me what it does"
  3. Fix issues
  4. Then write OA-02

Don't write five skills first and debug a wall of errors.

M4 · Tips

  • Stub all your skills as one-line descriptions first — see the whole map before committing to detail
  • Reference other skills by ID (see OA-02), never copy-paste
  • If Claude rewrites your skill into something bigger → push back. Terse beats comprehensive
  • Copy an existing skill (SI-01, AI-01) as a starting shape — blank pages kill momentum
Checkpoint: 3+ skill files in /skills/, all sections filled in, no TODO markers

MODULE 5 · 30 min

Write the Agent

Assemble the skills into a specialist

M5 · Purpose

Assemble your skills into an agent with:

  • A clear identity
  • A procedure (what to do when invoked)
  • A routing vocabulary (signals)

Output: agents/optimization_agent/ folder with README.md + system_prompt.md.

An Agent Is 6 Things

  1. Identity — who is this agent, what are they responsible for?
  2. Skills Table — your OA- skills, with when to use each
  3. Procedure — 4–5 numbered steps
  4. Output Format — what the agent's output looks like
  5. Halt Conditions — 3–4 conditions that stop the agent
  6. Rules — 3–4 rules about how the agent behaves

Reference: agents/supervisor/system_prompt.md · agents/demand_analyst/system_prompt.md

⚠ The Silent Failure

An agent only "knows about" skills listed in its Skills Table.

If you wrote OA-05 but forgot to list it → it will never run.

No error. No warning. Just silence.

Read your Skills Table aloud before moving on.

Your Agent's Three Signals

OPTIMIZATION_OPPORTUNITY

Excess stock, avoidable waste, a cheaper or faster alternative exists.

CRITICAL_WASTE_RISK

Near-term expiry, stranded stock, imminent loss if nobody acts.

NO_ACTION

Ran cleanly, nothing worth flagging. This is a valid answer, not an error.

Every signal must be tied to a decision rule in your system prompt: "emit X when Y".

M5 · Time Plan (30 min)

SectionWhat to writeMin

Procedure4–5 numbered steps referencing OA-0X12
Output FormatWhat the output looks like5

Common Pitfalls — Check Before Moving On

  • Skills Table lists a skill that doesn't exist as a file → silent failure
  • Agent folder named wrong (Optimization_Agent, optimisation_agent) → workflow in M6 won't resolve it
  • Procedure never says "emit signal X when Y" → agent finishes without emitting anything
  • Procedure step says "analyse stock" without naming a skill → agent won't know what to run

M5 · Tips

  • Write the system prompt as prose, not bullet salad. Instructions to a smart colleague, not a spec sheet
  • Procedure should read like a runbook — someone could follow it without knowing the domain
  • Halt conditions are how you stay safe. Be generous. A halt is cheap; a bad output isn't
  • Copy-paste demand_analyst/system_prompt.md and edit. Blank pages kill momentum

MODULE 6 · 15 min

Define Your Workflow

Wire your agent into the runner

M6 · Purpose

Register your agent in workflows/workflows.json so the runner can execute it.

Two things to add:

  1. Your three signals → global routing_signals array
  2. Your workflow WF-OA-01 → workflows object

Workflows Are the Orchestration Layer

  • They say which agent runs, in what order, with what task
  • They do not contain the intelligence — that lives in the agent and skills
  • Think of a workflow as a recipe, the agents/skills as ingredients

Add Your Signals (Globally)

At the top of workflows.json:

"routing_signals": [
  ... existing signals ...
  "OPTIMIZATION_OPPORTUNITY",
  "NO_ACTION",
  "CRITICAL_WASTE_RISK"
]
Signals must appear in two places:
1. The global routing_signals array (so the system recognises them)
2. Your agent's system prompt (so the agent knows when to emit them)

Add Your Workflow

"WF-OA-01": {
  "name": "Optimization Analysis",
  "steps": [
    {
      "agent": "optimization_agent",
      "task": "YOUR TASK DESCRIPTION — describe in plain English
               what the agent should analyse and what signals
               it should emit"
    }
  ]
}
The "agent" value must match your folder name in /agents/ exactly. Case-sensitive. Spelling-sensitive.

M6 · Tips

  • One missing comma breaks the whole file. Save and glance for red squiggles after every edit
  • Write "task" as a concrete instruction, not a vague mission:
    • ✅ "Analyse depot network for excess stock above the max-weeks policy; emit OPTIMIZATION_OPPORTUNITY when found"
    • ❌ "Do optimisation"
  • Ask Claude: "Is my workflows.json valid JSON?"
  • When you say "it won't run" in M7, first check JSON validity

MODULE 7 · 15 min

Run the Workflow Chain

The payoff

M7 · Purpose

Run your own agent end-to-end against your own study.

Read the output critically.

By the end: a file in outputs/ with at least one finding that came from a skill you wrote.

Before You Run — Verify

Check studies/<STUDY_ID>/data_drops/<today>/ contains:

  • rtsm_actuals.csv
  • ctms_plan.csv
  • erp_inventory.csv
  • site_inventory.csv (if your skills need it)

Missing? Regenerate from the Study Data Creation GUI.

Run It

/run-workflow-chain <STUDY_ID> WF-01 WF-OA-01

You'll see:

  1. Supervisor validates data (WF-01)
  2. Your Optimization Agent runs its analysis (WF-OA-01)

Design with Sonnet, run with Haiku. Saves tokens during execution.

"Completed" Doesn't Mean "Correct"

A workflow can complete with empty findings. Always open the output and read it.

Navigate to studies/<STUDY_ID>/outputs/<today>/ and read:

  • What inefficiencies did your agent detect?
  • What signal did it emit?
  • What actions did it recommend?

When the First Run Fails

It often does. The first thing to check:

  1. JSON validity in workflows.json (M6)
  2. Agent folder name matches the "agent" value (M5 / M6)
  3. Skill in the Skills Table exists as a file (M5)
  4. ID mismatches between CSVs and config (M2 root cause)

Read the error. Don't just re-run.

🎁 Bonus — Interrogate Your Agents

After the chain runs, ask questions in natural language:

  • "What happens if the study start date is delayed by 3 months?"
  • "What happens if study dispensing is paused for 8 weeks?"
  • "What happens if we close down site X?"
  • "What's the impact if I have to quarantine lot Y?"
  • "Clinical wants to add a new country using existing supplies. Give me a discovery-meeting checklist."
This is where the real insight surfaces. Don't skip it.
